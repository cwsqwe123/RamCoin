# AgentPay SDK version 0.3.0 release notes

**Release date:** 2026-04-05  
**Current version:** v0.3.0

These notes cover what is new in AgentPay SDK `v0.3.0` relative to `v0.2.0`.

For the baseline `x402` and `mpp` payment flows, Tempo session support, and prior chain/runtime additions, see [v0.2.0 release notes](./v0.2.0.md).

## Release summary

AgentPay SDK `v0.3.0` adds generic EIP-712 signing, improves MPP session handling, and fixes shared-config policy refresh so saved token-limit changes now reach the live daemon for existing wallets.

## What's new in v0.3.0

### EIP-712 signing

- added `agentpay sign-typed-data` for arbitrary EIP-712 typed-data signing through the local daemon policy path
- added `agentpay admin add-eip712-signing-policy` with `allow`, `manual-approval`, and `deny` modes scoped by network
- generic typed-data signing is evaluated separately from spend-limit accounting and defaults to manual approval unless policy allows it

### MPP and Tempo session workflows

- improved `agentpay mpp` session handling when servers return multiple payment challenges by selecting the correct MPP challenge
- added text-mode handling for `payment-need-voucher` stream control events, including automatic voucher and top-up follow-up
- added Tempo session demo and local server tooling with:
  - `pnpm example:mpp:session-demo`
  - `pnpm example:mpp:session-server`
- expanded local examples for persisted session open, reuse, and close flows
- fixed Tempo session open/top-up signing so the daemon signs the standard transaction payload hash instead of a Keychain-wrapped hash, restoring server-side verification

### Installer improvements

- Windows source installs now fail fast with a clearer JavaScript-only fallback hint
- runtime refresh continues to support wallet reuse through `agentpay admin setup --reuse-existing-wallet`

### Shared-config policy refresh

- fixed `agentpay admin token set-chain` so it no longer only updates `~/.agentpay/config.json`
- when reusable wallet metadata is present, saved per-token limit changes now immediately attempt to refresh the live daemon policy attachment for the existing wallet
- if that live apply step fails, the CLI restores the previous saved config instead of leaving the saved draft and live daemon out of sync
